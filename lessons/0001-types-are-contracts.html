Lesson 0001 · Brownfield baseline · 8 minutes

Types are contracts, not labels

Your win: predict which method an ordinary-looking call reaches before you trace into a large MVC 5 service layer.

Mission link: In brownfield C#, the type visible at the call site can matter as much as the object created at run time. Reading that distinction correctly prevents false control-flow assumptions.

The model

Every C# expression has a compile-time type. That type controls which members are available, which conversions are permitted, and which overload is selected. An object can also have a more specific run-time type, which matters for virtual dispatch and type tests.

PHP instinct

function describe($value) { /* inspect at runtime */ }
describe("paid");

The value arriving at run time dominates your mental model.

C# adjustment

void Describe(object value) { }
void Describe(string value) { }
object status = "paid";
Describe(status);

The call selects Describe(object) because status has compile-time type object.

Why this surprises people

Overload resolution occurs at compile time. It differs from overriding: an overridden virtual method is selected using the run-time object. Mature C# code often contains both mechanisms, so the first reading move is to identify the compile-time type at each call site.

Predict before clicking

static string Render(object value) => "object";
static string Render(string value) => "string";

object invoiceId = "INV-42";
Console.WriteLine(Render(invoiceId));

What is printed?

Brownfield reading move

  1. Find the variable, parameter, or property declaration.
  2. Record its compile-time type.
  3. List candidate overloads from that type context.
  4. Only then consider virtual dispatch on the run-time object.
